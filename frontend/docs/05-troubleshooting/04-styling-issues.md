# Styling Issues

This guide covers troubleshooting CSS, Tailwind, and styling problems in the CryoET Data Portal. Learn how to debug specificity conflicts, SSR hydration mismatches, Tailwind configuration issues, and more.

**Last updated:** December 10, 2025

## Tailwind CSS Issues

### Utility Classes Not Applying

#### Problem: Tailwind classes have no effect

**Symptom:**

```tsx
<div className="bg-blue-500 text-white p-4">
  {/* Styles don't apply */}
</div>
```

**Possible causes:**

1. **Class name typo**: Tailwind only includes classes it detects in your code
2. **Not in content path**: File isn't scanned by Tailwind
3. **Dynamic class names**: Tailwind can't detect dynamically constructed classes
4. **CSS specificity conflict**: Another style is overriding

**Solutions:**

**1. Check for typos:**

```tsx
// Wrong
<div className="bg-blue-50">  // Should be bg-blue-500

// Correct
<div className="bg-blue-500">
```

**2. Verify file is in content path:**

Check `tailwind.config.ts`:

```typescript
export default {
  content: ['./app/**/*.{ts,tsx,scss}'],
  // Your files must match this pattern
}
```

**3. Avoid dynamic class construction:**

```tsx
// Bad: Tailwind won't detect these classes
const color = 'blue'
<div className={`bg-${color}-500`}>

// Good: Use full class names
const colorClass = color === 'blue' ? 'bg-blue-500' : 'bg-red-500'
<div className={colorClass}>

// Better: Use style prop for truly dynamic values
<div style={{ backgroundColor: color }}>
```

**4. Check browser DevTools:**

Inspect the element to see if:
- Classes are present in the DOM
- Styles are applied but overridden (crossed out)
- Classes are missing entirely (not generated by Tailwind)

### Custom Tailwind Classes Not Working

#### Problem: Custom theme values don't apply

**Symptom:**

```tsx
// Using custom screen size from config
<div className="hidden screen-1345:block">
  {/* Doesn't respond to breakpoint */}
</div>
```

**Solution:**

Verify the custom value exists in `tailwind.config.ts`:

```typescript
export default {
  theme: {
    extend: {
      screens: {
        'screen-1345': '1345px', // Must be defined here
      },
    },
  },
}
```

**Restart dev server** after config changes:

```bash
# Kill and restart
pnpm dev
```

Tailwind's JIT compiler needs to pick up config changes.

### Tailwind Purging Necessary Classes

#### Problem: Classes work in development but missing in production

**Symptom:**

Styles work when running `pnpm dev` but break after `pnpm build`.

**Cause:**

Tailwind isn't detecting classes in certain files during the production build.

**Solution:**

**1. Ensure files are in content path:**

```typescript
// tailwind.config.ts
export default {
  content: [
    './app/**/*.{ts,tsx,scss}',
    // Add other locations if needed
    './components/**/*.{ts,tsx}',
  ],
}
```

**2. Use complete class names:**

```tsx
// Bad: May be purged
const spacing = 'p-4'
<div className={spacing}>

// Good: Full class name in JSX
<div className="p-4">
```

**3. Safelist critical classes:**

```typescript
// tailwind.config.ts
export default {
  content: ['./app/**/*.{ts,tsx,scss}'],
  safelist: [
    'bg-blue-500',
    'text-white',
    // Classes that are dynamically generated
  ],
}
```

## CSS Specificity Conflicts

### Styles Overriding Each Other

#### Problem: Styles don't apply as expected

**Symptom:**

```tsx
<div className="text-red-500" style={{ color: 'blue' }}>
  Text is blue, not red
</div>
```

**CSS Specificity Order** (lowest to highest):

1. Browser defaults
2. External stylesheets
3. Tailwind utilities
4. CSS Modules
5. Inline styles
6. `!important`

**Debugging specificity:**

1. **Inspect element in DevTools**
2. **Check Styles panel** - crossed-out styles are overridden
3. **Look at the right side** - shows which file/line has higher specificity

**Example conflict:**

```css
/* Component.module.css - Higher specificity */
.container {
  color: blue;
}
```

```tsx
/* Component.tsx - Lower specificity */
<div className={clsx(styles.container, 'text-red-500')}>
  {/* Blue wins because CSS Modules have higher specificity */}
</div>
```

**Solutions:**

**1. Use `!important` (last resort):**

```tsx
<div className="!text-red-500">
  {/* Forces red color */}
</div>
```

**2. Increase specificity with Tailwind:**

```tsx
<div className="[&]:text-red-500">
  {/* Arbitrary variant increases specificity */}
</div>
```

**3. Remove conflicting CSS Module style:**

```css
/* Component.module.css */
.container {
  /* Remove color rule */
  padding: 1rem;
}
```

**4. Use Tailwind merge utility:**

```typescript
import { twMerge } from 'tailwind-merge'

<div className={twMerge('text-red-500', dynamicClasses)}>
  {/* twMerge resolves conflicts by keeping the last class */}
</div>
```

### Material-UI Styles Conflicting

#### Problem: MUI styles override Tailwind

**Symptom:**

```tsx
import { Button } from '@mui/material'

<Button className="bg-blue-500">
  {/* Background doesn't change */}
</Button>
```

**Cause:**

MUI uses Emotion (CSS-in-JS) which has higher specificity than utility classes.

**Solution:**

**1. Use MUI's `sx` prop:**

```tsx
<Button
  sx={{
    backgroundColor: 'rgb(59 130 246)', // blue-500
    '&:hover': {
      backgroundColor: 'rgb(37 99 235)', // blue-600
    },
  }}
>
  Button
</Button>
```

**2. Use Tailwind with `!important`:**

```tsx
<Button className="!bg-blue-500">
  Button
</Button>
```

**3. Use design tokens from theme:**

```tsx
<Button sx={{ bgcolor: 'primary.main' }}>
  Button
</Button>
```

**Note:** The project prefers Tailwind over MUI's `styled()` API:

```typescript
// ESLint rule enforces this
'no-restricted-imports': [
  'error',
  {
    patterns: [
      {
        group: ['@mui/*'],
        importNames: ['styled'],
        message: 'Prefer tailwind over MUI styled components',
      },
    ],
  },
]
```

## CSS Modules Issues

### Type Definitions Not Generated

#### Problem: TypeScript errors for CSS Module imports

**Symptom:**

```tsx
import styles from './Component.module.css'
// Error: Cannot find module './Component.module.css'
```

**Cause:**

Type definitions (`.d.ts` files) for CSS Modules aren't generated.

**Solution:**

**Generate CSS Module types:**

```bash
# One-time generation
pnpm build:tcm

# Or watch mode
pnpm dev:tcm
```

This runs `typed-css-modules` which creates `.d.ts` files:

```
Component.module.css
Component.module.css.d.ts  ← Generated
```

**Verify generation:**

```typescript
// Component.module.css.d.ts (auto-generated)
export const container: string
export const header: string
```

**Add to `.gitignore`:**

```
*.module.css.d.ts
```

These files are generated and shouldn't be committed.

### Class Names Not Applying

#### Problem: CSS Module classes don't style the element

**Symptom:**

```tsx
import styles from './Component.module.css'

<div className={styles.container}>
  {/* No styles applied */}
</div>
```

**Debugging steps:**

**1. Check class name in CSS file:**

```css
/* Component.module.css */
.container {
  padding: 1rem;
}
```

**2. Verify import path:**

```tsx
// Correct
import styles from './Component.module.css'

// Wrong
import styles from './Component.css'  // Missing .module
```

**3. Inspect element in browser:**

Look for hashed class name like `Component_container__xJ9K1`. If missing, the style isn't loaded.

**4. Check file is in Tailwind content path:**

```typescript
// tailwind.config.ts
export default {
  content: ['./app/**/*.{ts,tsx,scss}'],  // Includes .scss
}
```

**5. Restart dev server:**

```bash
pnpm dev
```

### camelCase vs kebab-case

#### Problem: ESLint errors for CSS class names

**Symptom:**

```css
/* Component.module.css */
.some-class {  /* ESLint error: Use camelCase */
  padding: 1rem;
}
```

**Cause:**

Stylelint enforces camelCase for CSS Module classes:

```javascript
// stylelint config
{
  rules: {
    'selector-class-pattern': '^[a-z][a-zA-Z0-9]*$',
  }
}
```

**Solution:**

Use camelCase:

```css
/* Component.module.css */
.someClass {
  padding: 1rem;
}
```

```tsx
<div className={styles.someClass}>
```

## Server-Side Rendering (SSR) Issues

### Hydration Mismatch

#### Problem: React hydration error

**Symptom:**

```
Warning: Text content did not match. Server: "Loading..." Client: "Data loaded"
```

**Cause:**

Server renders one thing, client renders another.

**Common causes:**

1. **Using browser-only APIs during SSR:**

```tsx
// Bad: window is undefined on server
const width = window.innerWidth

// Good: Check if window exists
const width = typeof window !== 'undefined' ? window.innerWidth : 0
```

2. **Date/time formatting differences:**

```tsx
// Bad: Different timezone on server vs client
<div>{new Date().toString()}</div>

// Good: Format consistently
<div>{new Date().toISOString()}</div>
```

3. **Random values:**

```tsx
// Bad: Different random values
<div key={Math.random()}>

// Good: Stable keys
<div key={item.id}>
```

**Solution:**

Use `useEffect` for client-only code:

```tsx
import { useState, useEffect } from 'react'

function Component() {
  const [width, setWidth] = useState(0)

  useEffect(() => {
    setWidth(window.innerWidth)
  }, [])

  return <div>Width: {width}</div>
}
```

### Flash of Unstyled Content (FOUC)

#### Problem: Styles load after initial render

**Symptom:**

Page renders without styles briefly, then styles pop in.

**Cause:**

CSS isn't extracted for SSR or Emotion cache isn't configured.

**Solution:**

The CryoET Data Portal already configures Emotion SSR in `entry.server.tsx`:

```tsx
import createEmotionServer from '@emotion/server/create-instance'
import { createEmotionCache } from './emotion-cache'

const cache = createEmotionCache()
const { extractCriticalToChunks } = createEmotionServer(cache)

// Extract styles during SSR
const chunks = extractCriticalToChunks(markup)
```

If experiencing FOUC:

1. **Verify Emotion cache configuration**
2. **Check that critical CSS is inlined** in the HTML response
3. **Ensure Material-UI theme provider wraps the app**

## PostCSS and Build Issues

### PostCSS Syntax Errors

#### Problem: Build fails with PostCSS errors

**Symptom:**

```
Error: Failed to parse CSS
CssSyntaxError: Unclosed block
```

**Cause:**

Invalid CSS syntax, often from CSS nesting.

**Example:**

```css
/* Invalid nesting without plugin */
.container {
  .nested {  /* Requires postcss-nesting */
    color: red;
  }
}
```

**Solution:**

The project uses `postcss-nesting` plugin:

```javascript
// postcss.config.js
module.exports = {
  plugins: {
    'postcss-nesting': {},
    tailwindcss: {},
    autoprefixer: {},
  },
}
```

Ensure nesting syntax is correct:

```css
/* Correct nesting */
.container {
  & .nested {
    color: red;
  }

  &:hover {
    background: blue;
  }
}
```

### Autoprefixer Issues

#### Problem: Styles work in Chrome but not Safari

**Symptom:**

CSS features like Grid or Flexbox work in Chrome but break in Safari.

**Cause:**

Missing vendor prefixes for older browser support.

**Solution:**

Autoprefixer should handle this automatically. Verify it's configured:

```javascript
// postcss.config.js
module.exports = {
  plugins: {
    autoprefixer: {},
  },
}
```

Check browserslist configuration in `package.json`:

```json
{
  "browserslist": [
    "defaults",
    "not IE 11"
  ]
}
```

## Dark Mode Issues

The CryoET Data Portal doesn't currently implement dark mode, but if adding it:

### Tailwind Dark Mode

**Enable dark mode in config:**

```typescript
// tailwind.config.ts
export default {
  darkMode: 'class', // or 'media'
  theme: {
    extend: {
      colors: {
        // Define dark mode colors
      },
    },
  },
}
```

**Use dark mode classes:**

```tsx
<div className="bg-white dark:bg-gray-900">
  <p className="text-black dark:text-white">
    Content
  </p>
</div>
```

## Responsive Design Issues

### Breakpoints Not Working

#### Problem: Responsive classes don't apply at expected sizes

**Symptom:**

```tsx
<div className="hidden md:block">
  {/* Doesn't show at medium screens */}
</div>
```

**Debugging:**

1. **Check browser width:** Open DevTools → Responsive mode
2. **Verify breakpoint values:**

```typescript
// tailwind.config.ts
export default {
  theme: {
    extend: {
      screens: {
        'screen-1345': '1345px',
        // Custom breakpoints
      },
    },
  },
}
```

3. **Use standard Tailwind breakpoints first:**
   - `sm`: 640px
   - `md`: 768px
   - `lg`: 1024px
   - `xl`: 1280px
   - `2xl`: 1536px

**Mobile-first approach:**

Tailwind is mobile-first, so `md:block` means "block at md and above":

```tsx
{/* Show on mobile, hide on desktop */}
<div className="block md:hidden">
  Mobile content
</div>

{/* Hide on mobile, show on desktop */}
<div className="hidden md:block">
  Desktop content
</div>
```

### Container Max Width Issues

#### Problem: Content stretches too wide

**Solution:**

Use the configured content max-width:

```tsx
<div className="max-w-content mx-auto">
  {/* Content constrained to 1600px */}
</div>

<div className="max-w-content-small mx-auto">
  {/* Content constrained to 800px */}
</div>
```

Defined in `tailwind.config.ts`:

```typescript
maxWidth: {
  content: '1600px',
  'content-small': '800px',
}
```

## Debugging Styles in Browser

### DevTools Style Inspection

**Workflow:**

1. **Right-click element** → "Inspect"
2. **Styles panel** shows all styles affecting the element
3. **Computed panel** shows final computed values
4. **Hover over property** to see which rule applies it

**Useful features:**

- **Toggle style:** Click checkbox to disable/enable
- **Edit value:** Click value to modify
- **Add new style:** Click `+` to add rule
- **Force state:** Click `:hov` to force hover/focus states

### Finding Which File Contains a Style

**In DevTools Styles panel:**

Each rule shows the source file:

```
.container {
  padding: 1rem;
}
Component.module.css:12
```

Click the file name to open it in the Sources panel.

**For Tailwind classes:**

Styles show as coming from the Tailwind CSS bundle, not individual classes.

## Stylelint Errors

### Common Stylelint Issues

#### Problem: Linting fails with CSS errors

**Run Stylelint:**

```bash
pnpm lint:stylelint
```

**Auto-fix issues:**

```bash
pnpm lint:stylelint:fix
```

**Common errors:**

1. **camelCase class names:**
```css
/* Wrong */
.some-class { }

/* Right */
.someClass { }
```

2. **Property order:**
```css
/* Wrong: Properties not alphabetical */
.someClass {
  color: red;
  background: blue;
}

/* Right: Alphabetical order */
.someClass {
  background: blue;
  color: red;
}
```

3. **Missing units:**
```css
/* Wrong */
.someClass {
  width: 0;
  height: 0;  /* 0 doesn't need a unit */
}

/* Right */
.someClass {
  width: 0;
  height: 0;
}
```

## SDS (Science Design System) Integration

### Using SDS Tokens

The CryoET Data Portal integrates CZI's Science Design System tokens:

```typescript
// tailwind.config.ts
import sds from '@czi-sds/components/dist/tailwind.json'

export default {
  theme: {
    extend: {
      ...sds,  // Import SDS tokens
    },
  },
}
```

**Using SDS classes:**

```tsx
<h1 className="text-sds-header-xl leading-sds-header-xl">
  Title
</h1>

<p className="text-sds-body-m leading-sds-body-m">
  Body text
</p>
```

**SDS line heights:**

The config overrides faulty SDS line heights:

```typescript
lineHeight: {
  'sds-header-xxl': '34px',
  'sds-header-xl': '30px',
  // ...
}
```
